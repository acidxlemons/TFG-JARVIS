\section{OpenWebUI: Interfaz de Usuario Conversacional}
\label{sec:openwebui}
% ============================================================

OpenWebUI\footnote{\url{https://openwebui.com/}} es una plataforma de interfaz de usuario de código abierto diseñada para interactuar con modelos de lenguaje de forma conversacional a través del navegador web. Se trata de un proyecto mantenido por la comunidad que ha ganado una tracción significativa en el ecosistema de código abierto, acumulando más de 60.000 estrellas en GitHub y una comunidad activa de contribuidores \cite{openwebui2024docs}.

\subsection{Arquitectura interna de OpenWebUI}

OpenWebUI se estructura internamente como una aplicación web moderna con las siguientes capas:

\begin{itemize}
    \item \textbf{Frontend (SvelteKit):} Interfaz web reactiva construida con Svelte y SvelteKit que proporciona una experiencia de usuario similar a ChatGPT: entrada de texto con soporte Markdown, streaming de respuestas en tiempo real, gestión de conversaciones, carga de archivos adjuntos y un panel lateral de navegación entre chats. El frontend se comunica con el backend mediante API REST y WebSockets para el streaming.

    \item \textbf{Backend (Python/FastAPI):} Servidor API que gestiona la lógica de la aplicación: autenticación de usuarios, persistencia de conversaciones, gestión de modelos disponibles y enrutamiento de peticiones. El backend almacena toda la información de sesión en una base de datos SQL.

    \item \textbf{Base de datos (SQLite/PostgreSQL):} Almacena usuarios, conversaciones, mensajes, configuraciones y metadatos. En despliegues pequeños utiliza SQLite; en producción se recomienda PostgreSQL, opción elegida en este proyecto.
\end{itemize}

El factor decisivo fue el \textbf{sistema de Pipelines}: un mecanismo de extensión que permite inyectar lógica personalizada de procesamiento entre el usuario y el modelo de lenguaje \textbf{sin necesidad de modificar el código fuente de OpenWebUI}. Esta capacidad fue la que hizo posible construir todo el agente JARVIS como un plugin externo que se conecta a OpenWebUI mediante una API HTTP estandarizada, convirtiendo una interfaz genérica de chat en un asistente corporativo inteligente con capacidades RAG, scraping, OCR, consulta legislativa y más.

\subsection{Sistema de Pipelines de OpenWebUI}
\label{subsec:pipelines}

El sistema de Pipelines es el mecanismo central mediante el cual se extiende la funcionalidad de OpenWebUI. Funciona como un \textit{middleware} HTTP que intercepta los mensajes del usuario antes de que lleguen al modelo de lenguaje, permitiendo procesarlos, redirigirlos o transformarlos según la lógica definida por el desarrollador.

Desde el punto de vista arquitectónico, un Pipeline se despliega como un \textbf{servicio independiente} (típicamente un contenedor Docker) que expone una API HTTP. OpenWebUI se configura para enviar los mensajes del usuario a este servicio en lugar de directamente al modelo. El Pipeline procesa el mensaje, ejecuta la lógica necesaria (búsqueda en bases de datos, llamadas a APIs externas, etc.) y devuelve la respuesta al usuario a través de OpenWebUI.

El contrato que debe implementar un Pipeline se define mediante una clase Python con los siguientes métodos:

\begin{itemize}
    \item \texttt{\_\_init\_\_()}: Inicialización del pipeline, definición de nombre, identificador y configuración (\textit{Valves}).
    \item \texttt{on\_startup()}: Método ejecutado al arrancar el servicio (carga de recursos, conexiones).
    \item \texttt{on\_shutdown()}: Método ejecutado al detener el servicio (liberación de recursos).
    \item \texttt{pipe(user\_message, model\_id, messages, body)}: Método principal que recibe cada mensaje del usuario y devuelve la respuesta. Este es el \textbf{hook} central del sistema.
\end{itemize}

El parámetro \texttt{body} contiene metadatos adicionales como la información del usuario autenticado (email, nombre, rol, grupos), lo que permite implementar lógica de control de acceso personalizada.

Adicionalmente, OpenWebUI introduce el concepto de \textbf{Valves}: una clase \texttt{Pydantic BaseModel} anidada dentro del Pipeline que define parámetros configurables. Estos parámetros se exponen automáticamente en la interfaz de administración de OpenWebUI, permitiendo modificar la configuración del pipeline (URLs de backend, modelos, umbrales, mapeos de departamentos) sin necesidad de editar código ni reiniciar el servicio.

Este mecanismo de extensión fue la pieza clave que permitió construir todo el sistema JARVIS como una capa de inteligencia sobre OpenWebUI, manteniendo la interfaz de usuario intacta y delegando toda la lógica de negocio al pipeline y al backend propios.

\section{Ollama: Ejecución Local de Modelos de Lenguaje}
\label{sec:ollama}
% ============================================================

Ollama\footnote{\url{https://ollama.com/}} es un motor de ejecución de modelos de lenguaje diseñado para operar íntegramente en hardware local. Su función principal es gestionar la descarga, configuración, carga en memoria y ejecución de LLMs, exponiendo una API HTTP compatible con el estándar de OpenAI \cite{ollama2024docs}.

\subsection{Características técnicas}

Las características que hicieron de Ollama la herramienta idónea para mi proyecto son:

\begin{itemize}
    \item \textbf{Gestión dinámica de VRAM:} Ollama carga y descarga modelos de la GPU bajo demanda. Cuando la VRAM está ocupada por un modelo y se solicita otro, el primero se descarga automáticamente. Esta gestión resulta crítica cuando se operan simultáneamente modelos de texto, visión y embeddings en una única GPU con VRAM limitada.

    \item \textbf{Soporte de cuantización GGUF:} Soporte nativo para modelos en formato GGUF con múltiples niveles de cuantización (Q4\_0, Q4\_K\_M, Q8\_0), permitiendo ejecutar modelos que en precisión completa requerirían GPUs de datacenter en hardware de consumo \cite{ggmlGgufSpec}.

    \item \textbf{API compatible con OpenAI:} Expone endpoints de chat (\texttt{/v1/chat/completions}) y generación (\texttt{/api/generate}) compatibles con el formato de la API de OpenAI, lo que facilita su integración con herramientas del ecosistema.

    \item \textbf{Contenedorización nativa:} Disponible como imagen Docker oficial con soporte para NVIDIA Container Toolkit, lo que permite desplegar el motor de inferencia como un contenedor más dentro de la orquestación Docker Compose del sistema.
\end{itemize}

\section{Qdrant: Base de Datos Vectorial}
\label{sec:qdrant}
% ============================================================

Qdrant\footnote{\url{https://qdrant.tech/}} es un motor de búsqueda vectorial de código abierto (Apache 2.0) optimizado para operaciones de similitud sobre vectores de alta dimensionalidad. En el contexto de mi Trabajo de Fin de Grado, he utilizado Qdrant como el almacén principal de conocimiento del sistema RAG, donde indexo todos los fragmentos de documentos transformados en embeddings \cite{qdrant2024docs}.

\subsection{Características utilizadas}

\begin{itemize}
    \item \textbf{Índice HNSW:} Qdrant utiliza el algoritmo HNSW (\textit{Hierarchical Navigable Small World}) para indexar vectores y realizar búsquedas de vecinos próximos de forma eficiente. Este índice permite consultas sub-lineales incluso en colecciones con millones de puntos \cite{malkov2016hnsw}.

    \item \textbf{Búsqueda híbrida (Dense + Sparse):} Además de la búsqueda por similitud vectorial (densa), Qdrant soporta vectores dispersos que permiten implementar búsqueda léxica tipo BM25. La combinación de ambos enfoques constituye la estrategia de búsqueda híbrida del sistema, donde se recuperan fragmentos tanto por similitud semántica como por coincidencia exacta de términos \cite{qdrantHybridTutorial,qdrantSparseVectors}.

    \item \textbf{Filtrado por metadatos:} Cada punto vectorial almacena un \textit{payload} JSON con metadatos (nombre de archivo, página, fuente, departamento). Qdrant permite filtrar la búsqueda por estos campos, lo que habilita el control de acceso departamental: cada usuario solo consulta los documentos de las colecciones a las que tiene acceso.

    \item \textbf{Multi-colección:} El sistema define múltiples colecciones (\texttt{documents}, \texttt{documents\_CIVEX2}, \texttt{webs}) para organizar el conocimiento por origen y departamento, permitiendo búsquedas segmentadas.
\end{itemize}

\section{FastAPI: Framework del Backend}
\label{sec:fastapi}
% ============================================================

FastAPI\footnote{\url{https://fastapi.tiangolo.com/}} es un framework web de alto rendimiento para Python, basado en tipado estático y los estándares OpenAPI y JSON Schema. Lo seleccioné como base para desarrollar el backend de mi sistema por las siguientes razones \cite{fastapi2024docs}:

\begin{itemize}
    \item \textbf{Rendimiento asíncrono:} FastAPI se ejecuta sobre Uvicorn (servidor ASGI), lo que permite manejar peticiones concurrentes de forma nativa con \texttt{async/await}. Esto es especialmente relevante en un sistema donde las peticiones al LLM pueden tardar varios segundos.

    \item \textbf{Validación con Pydantic:} Los modelos de datos de entrada y salida (\texttt{ChatRequest}, \texttt{ChatResponse}, \texttt{SearchRequest}) se definen como clases Pydantic, garantizando validación automática de tipos, serialización y documentación.

    \item \textbf{Documentación OpenAPI automática:} FastAPI genera automáticamente documentación interactiva Swagger (\texttt{/docs}) y ReDoc (\texttt{/redoc}), facilitando la integración y depuración durante el desarrollo.

    \item \textbf{Ecosistema Python:} La implementación en Python permite integrar de forma natural todas las librerías de IA (SentenceTransformers, PaddleOCR, MSAL, Playwright) en un único servicio.
\end{itemize}

\section{LiteLLM: Proxy Unificado de Modelos}
\label{sec:litellm}
% ============================================================

LiteLLM\footnote{\url{https://docs.litellm.ai/}} es un proxy de código abierto que unifica las APIs de múltiples proveedores de modelos de lenguaje bajo una interfaz compatible con el formato de OpenAI. En este proyecto, he configurado LiteLLM para actuar como intermediario entre el backend y Ollama, ya que me proporciona capacidades avanzadas de gestión \cite{litellm2024docs}:

\begin{itemize}
    \item \textbf{Abstracción multi-modelo:} Cada modelo de Ollama se registra en LiteLLM con un alias descriptivo (\texttt{llama3.1-8b}, \texttt{llava}, \texttt{JARVIS}). El backend solicita modelos por su alias sin conocer los detalles de despliegue subyacentes.

    \item \textbf{Caché con Redis:} LiteLLM almacena en Redis las respuestas a consultas repetidas con un TTL configurable (1 hora), reduciendo la carga sobre la GPU y mejorando los tiempos de respuesta para consultas frecuentes.

    \item \textbf{Fallbacks automáticos:} Si el modelo principal (Q8\_0) falla o está ocupado, LiteLLM redirige automáticamente la petición al modelo ligero (Q4\_0) configurado como fallback.

    \item \textbf{Rate limiting:} Límites configurables por usuario y globales (solicitudes por minuto, tokens por minuto) para proteger los recursos de GPU.

    \item \textbf{Compatibilidad OpenAI:} Al exponer una API compatible con OpenAI, cualquier cliente o librería diseñada para OpenAI funciona directamente contra LiteLLM sin modificaciones.
\end{itemize}

\section{Docker y Orquestación de Contenedores}
\label{sec:docker}
% ============================================================

Docker\footnote{\url{https://www.docker.com/}} es la tecnología de contenedorización que he utilizado para empaquetar, desplegar y ejecutar todos los componentes de mi sistema de forma aislada y reproducible. Para facilitar la orquestación, utilizo Docker Compose, definiendo la totalidad de los 12 microservicios, sus dependencias, redes, volúmenes y recursos en un único fichero declarativo \texttt{docker-compose.yml}.

\subsection{Ventajas de la contenerización}

\begin{itemize}
    \item \textbf{Reproducibilidad:} El sistema completo se despliega ejecutando un único comando (\texttt{docker compose up -d}), garantizando que el entorno de producción sea idéntico al de desarrollo.

    \item \textbf{Aislamiento:} Cada servicio opera en su propio contenedor con sus dependencias específicas, evitando conflictos entre librerías (por ejemplo, la versión de CUDA requerida por PaddleOCR no interfiere con la de SentenceTransformers).

    \item \textbf{Red interna aislada:} Todos los servicios comparten una red Docker de tipo \textit{bridge} (\texttt{tfg-network}). Solo NGINX expone un puerto al exterior (8443), garantizando que servicios como PostgreSQL, Qdrant o Redis no sean accesibles desde fuera del servidor.

    \item \textbf{GPU passthrough:} Los servicios que requieren aceleración por GPU (Ollama, PaddleOCR) utilizan la directiva \texttt{deploy.resources.reservations.devices} con el runtime \texttt{nvidia-container-toolkit} para acceder a la GPU del host desde dentro del contenedor.
\end{itemize}

\section{Herramientas de Procesamiento}
\label{sec:herramientas-procesamiento}
% ============================================================

\subsection{PaddleOCR: Reconocimiento Óptico de Caracteres}

PaddleOCR\footnote{\url{https://github.com/PaddlePaddle/PaddleOCR}} es un motor de reconocimiento óptico de caracteres desarrollado por Baidu, capaz de detectar y transcribir texto en más de 80 idiomas. Se seleccionó para este proyecto por su rendimiento con aceleración GPU mediante CUDA, superando a alternativas como Tesseract en velocidad y precisión en documentos con layouts complejos \cite{paddleocr2024docs}.

El motor se ejecuta dentro de un contenedor Docker con acceso a la GPU y lo he integrado en el pipeline de ingesta: cuando el sistema detecta un PDF escaneado (sin texto seleccionable), cada página se convierte a imagen y se procesa con PaddleOCR para extraer el texto que posteriormente troceo, vectorizo y guardo en Qdrant.

\subsection{Playwright: Automatización de Navegadores}

Playwright\footnote{\url{https://playwright.dev/}} es un framework de automatización de navegadores desarrollado por Microsoft. A diferencia de soluciones basadas en peticiones HTTP simples, Playwright instancia un navegador Chromium completo en modo \textit{headless}, lo que permite renderizar páginas con JavaScript dinámico (React, Vue, Angular), resolver cargas diferidas (\textit{lazy loading}) y capturar el DOM final tras la ejecución de todos los scripts \cite{playwright2024docs}.

\subsection{Trafilatura: Extracción de Contenido Web}

Trafilatura es una librería de procesamiento del lenguaje natural especializada en extraer el contenido principal de páginas web, eliminando automáticamente elementos no informativos como menús de navegación, cabeceras, pies de página, barras laterales y publicidad. Actúa como primera opción en la cadena de extracción del scraper, con Readability y BeautifulSoup como alternativas de respaldo (\textit{fallback}).

\subsection{SentenceTransformers: Generación de Embeddings}

SentenceTransformers (Sentence-BERT) \cite{reimers2019sentence,sbertDocs} es la librería que utilizo para generar vectores densos de 384 dimensiones a partir de fragmentos de texto. El modelo específico que he empleado es \texttt{paraphrase-multilingual-MiniLM-L12-v2}, seleccionado por su soporte multilingüe (español e inglés), su tamaño moderado (420 MB) y su compatibilidad con ejecución en CPU. De esta forma, evito conflictos de VRAM con los modelos de IA generativa.

\section{Stack de Monitorización}
\label{sec:stack-monitorizacion}
% ============================================================

Para tener visibilidad sobre el funcionamiento del sistema, he integrado un stack completo de observabilidad compuesto por tres herramientas principales:

\subsection{Prometheus}

Prometheus\footnote{\url{https://prometheus.io/}} es un sistema de monitorización y alertas que recopila métricas temporales mediante \textit{scraping} activo de endpoints HTTP. En este proyecto, Prometheus scrapea el endpoint \texttt{/metrics} del backend FastAPI cada 15 segundos, almacenando series temporales de latencia, throughput, errores, distribución de modos de operación y uso de recursos.

\subsection{Grafana}

Grafana\footnote{\url{https://grafana.com/}} es la plataforma de visualización que consume las métricas almacenadas en Prometheus y las presenta en dashboards interactivos. Para este TFG, he diseñado y configurado tres dashboards principales:

\begin{itemize}
    \item \textbf{Dashboard de Backend:} Métricas de latencia (p50, p95, p99), peticiones por minuto, tasa de errores HTTP y distribución de modos de operación (RAG, chat, web search, scraping).
    \item \textbf{Dashboard de GPU:} Consumo de VRAM en tiempo real, temperatura del chip, utilización de CUDA cores y potencia consumida, capturados mediante NVIDIA DCGM Exporter.
    \item \textbf{Dashboard de Base de Datos:} Conexiones activas a PostgreSQL, queries por segundo y estadísticas de Qdrant.
\end{itemize}

\subsection{NVIDIA DCGM Exporter}

NVIDIA DCGM Exporter\footnote{\url{https://github.com/NVIDIA/dcgm-exporter}} es un exportador especializado que extrae métricas de rendimiento de las GPUs NVIDIA y las expone en formato compatible con Prometheus. Las métricas capturadas incluyen consumo de VRAM (GDDR6), utilización de los CUDA cores, temperatura del chip y potencia. Estas métricas son fundamentales para detectar cuellos de botella en la inferencia y gestionar la carga de modelos.

\section{Otras Tecnologías del Ecosistema}
\label{sec:otras-tecnologias}
% ============================================================

\begin{itemize}
    \item \textbf{PostgreSQL 16 con pgvector:} Base de datos relacional que almacena usuarios, conversaciones, mensajes y metadatos de OpenWebUI. La extensión pgvector se mantiene instalada como capacidad futura para operaciones vectoriales secundarias.

    \item \textbf{Redis 7:} Almacén en memoria utilizado como caché por LiteLLM para respuestas repetidas. Cada entrada tiene un TTL configurable tras el cual se invalida automáticamente.

    \item \textbf{MinIO:} Servidor de almacenamiento de objetos compatible con Amazon S3. Preserva una réplica aislada de todos los documentos binarios del sistema, proporcionando un mecanismo de backup independiente de la base de datos.

    \item \textbf{NGINX:} Proxy inverso que actúa como punto de entrada único del sistema. Gestiona la terminación TLS/SSL, la mejora de protocolo a WebSocket (necesaria para el streaming de tokens) y la redirección interna de peticiones a OpenWebUI.

    \item \textbf{MSAL (Microsoft Authentication Library):} Librería de autenticación de Microsoft utilizada tanto para el flujo SSO con Azure AD (OpenID Connect) como para la conexión a SharePoint mediante el flujo de credenciales de cliente (\textit{Client Credentials}) de OAuth 2.0 \cite{msgraph2024docs}.
\end{itemize}
